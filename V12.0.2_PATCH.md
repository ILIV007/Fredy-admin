## 1. Summary

This is a **precision optimization patch** — NOT a scheduler redesign.
The v12.0.0 Three-Layer Cron Architecture and v12.0.1 Quiet Hours Guard
remain structurally unchanged. Three specific issues are fixed:

| Issue | Problem | Fix |
|-------|---------|-----|
| 1. Random Jitter not exact | 10-min tolerance caused early publish (17:20 instead of 17:24) | Removed tolerance — `now >= scheduledTime` (exact) |
| 2. Quiet hours still write KV | `QUIET_SKIP_KEY` wrote 1 KV entry per quiet tick | Removed KV write entirely — `console.log` only |
| 3. Provider refresh blind | Layer 2 refreshed even when queues were full | Smart Sleep — skip if quiet AND all queues >= min |

## 2. Issue 1 — EXACT scheduledTime Trigger

### Before (v11.18.0–v12.0.1)
```typescript
const CRON_TOLERANCE_MINUTES = 10;
if (nowMinutes < scheduledMin - CRON_TOLERANCE_MINUTES) {
  continue; // skip
}
```
This fired UP TO 10 minutes BEFORE `scheduledTime`. Dashboard said
"17:24" but actual publish was "17:20" — a mismatch.

### After (v12.0.2)
```typescript
// EXACT — no tolerance
if (nowMinutes < scheduledMin) {
  continue; // skip
}
```
The scheduler fires ONLY when `now >= scheduledTime`. With a 20-min
cron, the actual publish lands on the first tick AT OR AFTER
scheduledTime (0-20 min delay).

### Example
```
Window: 16:00-18:00
scheduledTime: 17:24

17:20 tick → 17:20 < 17:24 → WAIT
17:40 tick → 17:40 >= 17:24 → PUBLISH (16 min delay)

Active hours example:
scheduledTime: 09:15
09:00 tick → WAIT
09:20 tick → PUBLISH (5 min delay)
```

### Files Modified
- `src/services/scheduler-service.ts` — `findDueSlot()`: removed `CRON_TOLERANCE_MINUTES`
- `src/entry/manager.ts` — scheduler debug API: pending/dueNow now uses exact `schedMin` (no `-10`)

## 3. Issue 2 — Zero-KV Quiet Hours

### Before (v12.0.1)
During quiet hours, the guard wrote `QUIET_SKIP_KEY` to KV every 20
minutes (overwrite-in-place, 1h TTL). This was ~22 writes/day —
better than the old design but still not zero.

### After (v12.0.2)
```typescript
if (isQuiet) {
  console.log(JSON.stringify({ ... }));  // console ONLY
  return;  // 0 KV reads, 0 KV writes
}
```
**Zero KV operations during quiet hours.** The dashboard computes
quiet-hours status LIVE from settings + current time
(`cronOptimization.currentState`) — no stored marker needed.

### KV Usage Comparison

|------|---------|---------|
| Quiet-hours tick | 0 reads + 1 write | **0 reads + 0 writes** |
| 22 quiet ticks/day | 22 writes | **0 writes** |

## 4. Issue 3 — Provider Refresh Smart Sleep

### Before (v12.0.0–v12.0.1)
Layer 2 ran every 2 hours unconditionally. Even if all queues were
full and it was quiet hours, it still acquired the lock, ran
`maintainQueue()`, and called `refreshDueProviders()`.

### After (v12.0.2)
```typescript
// Smart Sleep check (before acquiring lock)
if (isQuiet && allQueuesAtOrAboveMin) {
  console.log({ status: "smart_sleep", reason: "quiet_hours_and_queues_full" });
  return;  // No lock, no API calls, no Gemini
}
```

**Two-level optimization:**
1. **Top-level Smart Sleep**: If quiet hours AND all queues >= min →
   skip the ENTIRE tick (no lock, no API calls). Only 2 KV reads
   (settings + queue depth).
2. **Provider engine skip**: If not quiet (or queues are low) but
   `maintainQueue()` found all queues OK → skip
   `refreshDueProviders()` (no point fetching new content when full).

### Logic
```
Layer 2 fires (every 2h)
  |
  ├── Is it quiet hours?
  |     ├── YES → Are ALL queues >= min?
  |     |         ├── YES → SMART SLEEP (return, 0 API calls)
  |     |         └── NO  → continue (prepare content for next active window)
  |     └── NO  → continue
  |
  ├── Acquire lock
  ├── maintainQueue() — refill low queues
  ├── Was any queue low?
  |     ├── YES → refreshDueProviders() (fetch new content)
  |     └── NO  → skip provider engine (queues already full)
  └── Release lock
```

### Files Modified
- `src/entry/cron-providers.ts` — added Smart Sleep check before lock; conditional provider engine refresh

## 9. Execution Flow (v12.0.2)

```
Layer 1 (every 20 min):
  cron trigger
    → read settings (cached)
    → scheduler disabled? → return
    → maintenance mode? → return
    → QUIET HOURS? → console.log ONLY → RETURN (0 KV ops)
    → processScheduledQueue
    → scheduler.tick()
      → read plan (1 KV read)
      → findDueSlot: now >= scheduledTime? (EXACT, no tolerance)
      → if due → publish (dedup + AI + image + Telegram + history)
    → write lastLog (only if work done)

Layer 2 (every 2h):
  cron trigger
    → read settings
    → SMART SLEEP CHECK:
      quiet hours AND all queues >= min?
      → YES → console.log → RETURN (no lock, no API calls)
      → NO  → continue
    → acquire lock
    → maintainQueue (refill low queues)
    → any queue was low?
      → YES → refreshDueProviders (fetch new content)
      → NO  → skip provider engine (queues full)
    → release lock

Layer 3 (every 24h):
  cron trigger
    → generate tomorrow's plan
    → cleanup old KV
    → flush stats
```
